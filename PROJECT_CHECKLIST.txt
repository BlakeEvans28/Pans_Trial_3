═══════════════════════════════════════════════════════════════════════════════
 KEY DESIGN DECISIONS
═══════════════════════════════════════════════════════════════════════════════

1. ENGINE INDEPENDENCE
   WHY: Allows headless testing, different UIs, AI integration
   HOW: No pygame in engine/

2. ACTION-BASED COMMUNICATION
   WHY: Clean separation, easy to validate, natural for AI
   HOW: All moves are Action objects

3. TOROIDAL GRID
   WHY: Matches game rules (wrapping edges)
   HOW: Use modulo arithmetic for position normalization

4. TYPE HINTS EVERYWHERE
   WHY: Better IDE support, catches errors early, documents code
   HOW: Annotate all parameters and returns

5. COMPREHENSIVE TESTING
   WHY: Catches regressions, validates rules
   HOW: Unit tests for each system

 COMMON TASKS GOING FORWARD

Add a new feature?
  1. Modify engine/ if it's game logic
  2. Modify ui/ if it's rendering
  3. Add tests in tests/
  4. Run: pytest tests/test_rules.py -v

Add a new card effect?
  1. Add to _apply_card_effect() in game_state.py
  2. Add test in tests/test_rules.py
  3. Add rendering in board_renderer.py

Add an AI agent?
  1. Create class with choose_action(state, player_id) method
  2. Query state using public API
  3. Return valid action from get_legal_moves()
  4. Run with engine (no modifications needed)
